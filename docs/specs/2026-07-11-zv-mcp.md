# Spec: `zv mcp` - MCP server for driving the FragForge desktop app from coding agents

Status: draft v1 (2026-07-11).
Owner: Luis.

## Problem

Coding agents (Claude Code, Codex) cannot drive the FragForge desktop app today.
The `zv` CLI works for humans in a shell, and `zv tui` works interactively, but agents need a stable, typed, discoverable tool surface with machine-readable errors.
The desktop app (FragForge Studio) is an Electron shell around `zv-orchestrator` plus the Next.js web UI, both on loopback, so the orchestrator HTTP API is the single programmatic seam.
The primary deployment is an agent running on the same Windows PC as FragForge Studio, driving the running app; because that machine has CS2, HLAE, and a GPU, the full pipeline including record and render is in scope, not just the parse-only subset.

## Goal

A new `zv mcp` subcommand that starts an MCP server over stdio, exposing the demo-to-Short pipeline as MCP tools.
By default it finds the running desktop app by itself (see Connection discovery), so an agent on the same machine registers it with one line:

```jsonc
// .mcp.json / Claude Code config
{ "mcpServers": { "fragforge": { "command": "zv", "args": ["mcp"] } } }
```

## Non-goals (v1)

- Stream-clip jobs (Twitch/VOD flow): v2.
- Remote/authenticated topologies beyond passing an existing `ZV_MUTATION_TOKEN` through: the design must not preclude them, but v1 targets loopback.
- MCP resources/prompts: tools only in v1.
- Any server-side changes to the orchestrator API.

## Architecture

`zv mcp` is a pure thin client of the orchestrator HTTP API, exactly like `zv tui`.
It reuses `internal/tuiclient` for every call and imports no server-side domain packages.
The MCP layer is a new package `internal/mcpserver`, kept thin in `cmd/zv` per the repo rule that `cmd/` entrypoints stay thin.

SDK: `github.com/modelcontextprotocol/go-sdk` (the official Go SDK), stdio transport.

### Connection discovery

The desktop app does not listen on a fixed port: Electron picks a per-install stable loopback port per service and persists it in `<userData>/ports.json` as `{"orchestrator": <port>, "web": <port>}` (`desktop/src/main.ts`, `stablePort`).
Because the primary target is the desktop app, discovering that port is core v1 behavior, not a fast follow.
Resolution order:

1. `--url` flag.
2. `$ORCHESTRATOR_URL`.
3. Read `ports.json` from the Electron `userData` dir and use `http://127.0.0.1:<orchestrator>`. On Windows that dir is `%APPDATA%\fragforge-studio` (Electron derives `app.name` from the package.json `name` field; `productName` only exists under the electron-builder `build` key, so it does not affect the path - verify once on a real install).
4. Fall back to `http://127.0.0.1:8080` (a dev `zv serve`).

After resolving, probe `GET /healthz`; if it fails, the server still starts (MCP clients launch it eagerly) but every tool returns `orchestrator_unavailable` with a message telling the agent to launch FragForge Studio.
`--token` / `$ZV_MUTATION_TOKEN` is forwarded as `X-FragForge-Token` when set (only needed on non-loopback binds; the desktop app binds loopback without a token).

### Getting `zv` onto the Windows PC

The desktop installer ships `zv-orchestrator.exe` but not `zv.exe`, and agents on the user's PC need the latter to run `zv mcp`.
v1: build and publish `zv.exe` as a release artifact (it is already a static CGO-off binary) and document dropping it next to the agent config.
Fast follow: have `desktop/scripts/assemble.mjs` bundle `zv.exe` into the install dir, so the MCP registration can point at a well-known path inside the FragForge Studio installation.

## Tool surface (v1)

Tool names use snake_case; every tool returns structured JSON content.
The mapping is 1:1 onto `internal/tuiclient` methods, plus one synthesized tool (`next_step`) that ports the TUI reconciler.

| Tool | tuiclient method | Notes |
|---|---|---|
| `get_capabilities` | `Capabilities` | Which stages are configured (capture is Windows-only); agents must call this before offering record/render. |
| `list_presets` | `Presets` | Render variants and recording presets. |
| `list_jobs` | `ListJobs(limit)` | Summary list. |
| `get_job` | `GetJob(id)` | Full job status. |
| `create_job` | `CreateJob(demoPath, targetSteamID)` | Uploads a local `.dem`; empty target starts the roster-scan flow. |
| `get_roster` | `GetRoster(id)` | Player table (K/D/A, HS%, ADR, rating) for target selection. |
| `start_parse` | `StartParse(id, targetSteamID)` | Assigns the target to a scanned job. |
| `get_plan` | `GetPlan(id)` | Kill plan. |
| `get_moments` | `GetMoments(id)` | Scored clip candidates. |
| `start_recording` | `StartRecording(id, preset, segmentIDs)` | Gated on capabilities; never auto-retried (GPU minutes). |
| `start_compose` | `StartCompose(id)` | |
| `start_render` | `StartRenderVariant(id, variant)` | |
| `get_render` | `GetRenderVariant(id, variant)` | |
| `download_final` | `DownloadFinal(id, dst)` | Writes the MP4 to a caller-supplied local path and returns the path, never the bytes. `DownloadFinal` takes an `io.Writer`, so the tool handler owns creating the file. |
| `next_step` | `NextStep(status, render)` | The one-action-per-job reconciler; see below. |

### `next_step` is the agent's steering wheel

`internal/tuiclient/reconcile.go` already reduces the job state machine (`queued -> scanning -> scanned -> parsing -> parsed -> recording -> recorded -> composing -> composed -> done | failed`) to exactly one primary action (`pick-target`, `record`, `render`, `retry`, `wait`, ...).
The `next_step` tool fetches the job plus its render state and returns `{step, actionable, label, suggested_tool, suggested_args_hint}` so an agent can loop `next_step -> act -> poll` without re-deriving the state machine in its prompt.
This mirrors how `zv tui` drives the pipeline and is the main thing that makes the server agent-friendly rather than a raw API mirror.

### Long-running stages and polling

MCP tools must return promptly; recording and rendering take minutes.
All `start_*` tools return immediately after enqueue with the job snapshot.
Agents poll with `get_job` / `get_render` / `next_step`.
Each `start_*` response embeds a `poll_hint_seconds` field (parse ~2s, record/render ~10s) so agents do not hammer the API.

## Input/output schemas

Input schemas are generated from Go structs with `jsonschema` tags (the official SDK supports this), one request struct per tool in `internal/mcpserver/tools.go`.
Output reuses the DTOs in `internal/tuiclient/types.go` serialized as JSON, so the MCP surface never forks from what the TUI sees.
Every tool description states preconditions in one sentence (for example `start_recording` requires job status `parsed` and capture capability), because tool descriptions are the only documentation the agent reliably reads.

## Error model

`tuiclient.APIError` maps to MCP tool errors (`isError: true`) with a structured body: `{status_code, code, message, retryable}`.
Specific mappings:

- Orchestrator unreachable: `code: "orchestrator_unavailable"`, message tells the agent to check that FragForge Studio (or `zv serve`) is running, `retryable: true`.
- `IsNotReady` (409-style stage precondition): `code: "not_ready"`, include the current job status and the `next_step` result so the agent self-corrects.
- `IsNotFound`: `code: "not_found"`.
- Capability missing on `start_recording`/`start_render`: fail fast client-side with `code: "capability_missing"` before hitting the API.

## Security

- v1 assumes loopback: no token needed, matching the orchestrator's own default.
- If `--token`/`$ZV_MUTATION_TOKEN` is set, forward it on every call (tuiclient already does).
- `create_job` and `download_final` touch the local filesystem; paths come from the agent, which already has filesystem access on the same machine, so no extra sandboxing is added.
- The server never logs demo contents or tokens; logs go to stderr (stdout is the MCP transport and must stay clean).

## Testing

- Unit: table tests for the tool handlers against a `httptest` orchestrator stub, reusing the fixtures style of `internal/tuiclient/client_test.go`.
- Contract: one round-trip test per tool asserting the JSON schema of inputs and the shape of outputs.
- E2E: a script that starts `zv serve` with `ZV_DATABASE_URL=memory`, connects an MCP client over stdio, and walks upload -> roster -> parse -> plan on a fixture `.dem`; record/render are asserted only up to the capability gate on Linux.

## Live visibility in the desktop UI (milestone 5)

The point of targeting the desktop app is that agent actions show up graphically in the open app, the way Blender MCP actions show up in the Blender viewport.
Driving the orchestrator alone does not achieve that today, because the Studio web UI never enumerates server-side jobs:

- The matches page (`web/app/(app)/matches/page.tsx`, the screen Electron loads) calls `listMatches()`, which is hardcoded to return an empty array (`web/lib/api/real.ts`).
- The reel library (`/videos`) renders only `ReelIntent` records persisted in browser `localStorage` (`web/lib/api/reel-store.ts`); intents are created exclusively by the UI's own forge flow, and the poll loop only advances jobs it already has intents for.
- There is no list proxy route under `web/app/api/demos/`, even though the orchestrator already exposes `GET /api/jobs`.

So an MCP-created job would run the whole pipeline while the open app shows nothing.
Milestone 5 closes that gap:

1. Add a list proxy route (`GET /api/demos`) in the web app forwarding to the orchestrator's `GET /api/jobs` through the existing `callOrchestrator` helper.
2. Implement `listMatches()` for real on top of it, and give the matches page a poll loop (reuse `web/lib/poll-loop.ts` cadences) so externally-created jobs appear within a couple of seconds and advance live.
3. Reconcile the reel library with the server: derive a synthetic intent for any server job with render activity that has no local intent, so MCP-triggered renders appear in `/videos` and are advanced by the existing reconcile loop. Locally-created intents keep precedence; synthesis must never duplicate or clobber them.
4. E2E: with the web app and a memory-mode orchestrator running, create a job via `zv mcp` (or plain HTTP) and assert it becomes visible on `/matches` without any UI-side action (Playwright CLI, headless chromium).

## Milestones

1. `internal/mcpserver` skeleton + `zv mcp` wiring, read-only tools (`list_jobs`, `get_job`, `get_capabilities`, `list_presets`, `get_roster`, `get_plan`, `get_moments`, `next_step`).
2. Mutating tools (`create_job`, `start_parse`, `start_recording`, `start_compose`, `start_render`, `download_final`) with the error model.
3. Desktop discovery (`ports.json`) + E2E test + a `docs/` snippet showing Claude Code and Codex registration on Windows.
4. `zv.exe` shipped in the desktop install dir via `assemble.mjs`.
5. Live visibility in the desktop UI (see section above).
6. (v2) stream-clip tools, remote/authenticated topology.
