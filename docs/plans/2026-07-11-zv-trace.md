# Plan: `zv trace` - headless JSON-mode QA for the pipeline

Date: 2026-07-11.
Branch: `feat/zv-trace`.
Goal: inspect and regression-test the pure decision layers of the pipeline (parse -> moments -> selection -> render plan) in seconds, with no CS2/HLAE, no FFmpeg execution, no browser, and minimal RAM.

## Motivation

Every expensive stage of the pipeline derives from cheap, pure decisions: the kill plan, the moment scores, the chosen segments, and the FFmpeg argv that would run.
If those roots are correct, the high-level output is correct almost by construction.
Today those layers are only observable piecemeal (`zv short --dry-run`, `zv-parser parse`, `--fake` recorder), and there is no single artifact that captures every decision for a given input, nor golden tests over it.

## Deliverable

A new in-process subcommand `zv trace` plus an `internal/trace` package.

### Inputs

- `--demo <path> --steamid <id>`: full trace starting from a real `.dem` (uses `internal/parser`).
- `--from-plan <killplan.json>`: skip parsing and trace from an existing kill plan.
  This is the key testability seam: fixtures can be committed as JSON, no `.dem` needed in the repo.
- `--rules <file>`: optional rules JSON, same semantics as `zv-parser parse`.
- `--segment-mode kills|smokes|utility`: same values as the parser.
- `--preset <name>`: editor render preset (default same as `zv short`).
- `--out <path|->`: output, default stdout.
- `--pretty`: indented JSON.
- `--deterministic`: fixed job UUID, omitted timestamps, placeholder `ffmpeg` path.
  Golden tests always use this flag; interactive use gets real values by default.

### Output: one JSON trace document

```
TraceDocument {
  schema_version        int
  source                { kind: "demo"|"plan", path, sha256 (demo only) }
  killplan              killplan.Plan          // the parse decision
  moments               moments.Document       // scoring decision (includes per-moment Score)
  selection             { segment ids chosen for recording, ordering, rationale fields already present in moments }
  render                { preset name, output shape, edit document summary, ffmpeg_argv [][]string }
  warnings              []string
}
```

Reuse the existing JSON-tagged types verbatim (`killplan.Plan`, `moments.Document`); do not invent parallel structs for data that already has a contract.
`render.ffmpeg_argv` comes from the pure builders in `internal/editor/ffmpeg.go` (`BuildFFmpegCommand` and friends), never from executing FFmpeg.

## Architecture and seams (verified in code)

- Parse: replicate the flow of `cmd/zv-parser/app.go` `runParse` (lines 91-175): open demo, sha256, `demoinfocs.NewParser`, `parser.RunWithOptions(p, steamid, rules, meta, RunOptions{SegmentMode})`.
  Factor the shared logic into `internal/trace` only if it does not require touching `cmd/zv-parser`; otherwise duplicate the thin wiring (it is ~30 lines) and keep `zv-parser` untouched.
- Moments: `moments.Build(jobID uuid.UUID, plan killplan.Plan) Document` (internal/moments/moments.go:77). Pure.
- Render plan: follow what `cmd/zv/short_command.go` `resolveShortPlan` (line 131) does to resolve preset and output spec, then build the FFmpeg argv with `editor.BuildFFmpegCommand` using a `ShortEdit` assembled the same way `zv-editor` would, with placeholder input clip paths (one synthetic clip path per selected segment, e.g. `segment-<id>.mp4`).
  The point is to expose the argv as data; paths are placeholders by construction in a trace.
- Command wiring: new `case "trace"` in `cmd/zv/app.go` dispatching to `cmd/zv/trace_command.go`, in-process (no delegate binary).
  `zv trace` must appear in the top-level help output.

## Testing

1. Unit tests in `internal/trace`: build a small `killplan.Plan` in Go, run the trace, assert structural invariants (segments propagate to moments, argv non-empty, deterministic mode stable).
2. Fixture: `testdata/trace/killplan.sample.json`, a realistic hand-written kill plan (2-3 segments, multiple kills, utility) that loads through the real `killplan` types.
3. Golden test: run `zv trace --from-plan <fixture> --deterministic --pretty`, compare byte-for-byte against `testdata/trace/golden/trace.sample.json`; regenerate with `go test ./internal/trace/... -run TestTraceGolden -update` (standard `-update` flag pattern; the flag only exists in that package).
4. Real-demo E2E: gated on `ZV_E2E_DEMO` exactly like the existing parser E2E; skips when the demo is absent.
   No `.dem` is committed.

## Execution plan (agents)

VPS constraints apply: builds and test runs are heavy, run strictly one at a time, wrapped in `systemd-run --scope -q -p MemoryMax=1200M nice -n 10` using host Go at `/usr/local/go/bin/go`.

- Task A (Opus): `internal/trace` package + `cmd/zv/trace_command.go` + dispatch wiring + unit tests. Runs alone, builds included.
- Task B (Sonnet, after A): kill plan fixture, golden test with `-update` flag, `ZV_E2E_DEMO`-gated E2E, help-text check. Builds included, serialized after A.
- Task C (Haiku, parallel with B, no builds): document the trace tool in the repo `CLAUDE.md` pipeline section, following its existing style.

## Verification plan

1. Full `go build ./...` and `go test ./...` (memory-capped) on the final tree.
2. Manual smoke: `zv trace --from-plan testdata/trace/killplan.sample.json --pretty | jq` and inspect the argv and selection by hand.
3. Two adversarial reviews with fresh context (verifier agent), different angles:
   - Angle 1: correctness and repo-convention consistency of the new code (types reused vs duplicated, flag semantics vs sibling commands, error handling and exit codes).
   - Angle 2: testing and determinism (golden stability across runs, fixture realism, what the tests would fail to catch, exercising the built binary directly).
4. Final review of the whole diff by the main session, fix findings, re-verify, then commit, push, and open a draft PR.

## Non-goals

- No simulation mode for the orchestrator or the web UI.
- No changes to recorder, composer, editor execution paths.
- No committed `.dem`.
- No new scoring logic; the trace only exposes existing decisions.
